### 1.1 핵심 원칙 (채택)

1. **결과 먼저, 과정은 접기** — 상단·요약에는 *지금 차량이 어떤 상태인가*만 단정적으로 보여 준다. 추론·Timeout·API 필드명은 기본 숨김.
2. **불안을 키우는 기술 단어 금지(1차 화면)** — `추론`, `configSynced`, `fleet_telemetry_config`, `Baseline`, `TELEMETRY`, `ASLEEP`, `READY` 등을 뱃지/헤드라인에 그대로 쓰지 않는다.
3. **신뢰는 “숨김”이 아니라 “쉬운 확정 + 필요할 때만 근거”** — 절전/주차 판정이 신호 공백 기반이어도 UI는 「주차(절전)」처럼 확정 표현. 근거는 운영·연동 섹션 또는 「자세히」에만.
4. **조치 가능한 경고만 1차 노출** — 스트림이 정상인데 Tesla `synced`만 false인 경우처럼 **운영상 문제 없는 상태**는 경고 톤을 쓰지 않거나 정보 톤으로 내린다.
5. **VD-1과의 관계** — VD-1은 null/추론을 *숨기지 말고 신뢰 가능하게* 보여 주라고 했다. 본 문서는 그 정보를 **삭제하지 않고**, **관리자 언어로 옮기고 계층을 나눈다** (요약=업무 언어, 운영=기술 근거).

## 2. 문제 정의 (초안 분석)

### 2.1 초안에서 잘 짚은 점 (채택)

| 초안 | 평가 |
|------|------|
| 일부 문구가 **시스템 내부 논리**에 치우침 | **동의** — 법인 관리자 관점과 어긋남 |
| `취침 (추론)` + `Telemetry 공백으로 취침을 추론…` | **동의** — ‘추론’이 **부정확·대충**으로 읽힘 |
| 백엔드 공백 감지 로직은 유지하되 UI에 과정 과시 불필요 | **동의** |
| 수정안: `주차 (절전)` / `마지막 신호: 4시간 전 (절전 모드 진입)` | **방향 채택** — 표현은 §4에서 확정 |

### 2.2 보완 관점

- 대상 사용자는 “일반 소비자 앱”이 아니라 **법인 플릿 관리자**다. 전문 용어 전면 금지가 아니라 **1차 화면에서 업무 용어**, **2차(운영)에서 기술 용어**.
- “확정적 표현”이 **허위 확정**이 되면 안 된다. 예: 신호 없이 장시간이면 「주차(절전)」는 OK, 「차량이 잠겼고 안전하다」까지 단정하면 안 됨.
- 동일 원칙을 **연동 경고·타임라인·버튼·출처 힌트**에도 적용해야 한다. (초안 예시 외 As-Is 다수)

## 3. As-Is — 개발자 언어로 보이는 문구 (상세 화면)

수용 VIN·최근 온보딩 기준으로 관리자가 불편/오해하기 쉬운 항목.

| 구역 | As-Is 예 | 관리자 관점 문제 |
|------|----------|------------------|
| Summary 상태칩 | `취침 (추론)` | ‘추론’=불확실·대충 |
| Summary 보조 | `Telemetry 공백으로 취침을 추론했습니다 · N시간 전` | 영문 제품명 + 내부 알고리즘 노출 |
| 상태 설명 카드 | `관제 준비(READY)이며 취침은 Telemetry 공백으로 추론…` | 영문 enum 병기, 문장 과밀 |
| 상단 경고 | 뱃지 `configSynced false` · `fleet_telemetry_config synced` | 필드명 노출, 스트림 정상인데도 **고장처럼** 보임 |
| 연동 타임라인 | `Telemetry config` / `미동기(configSynced false)` / 회색 | 미완료·오류로 오해 (실제는 비동기 채택 대기일 수 있음) |
| 타임라인·버튼 | `Baseline`, `Wake`, `Telemetry 다시 연결` | Baseline=제원 스냅샷임을 모름 |
| 구독 필드 | `활성 · config 미동기` | “미동기”=연동 실패로 읽힘 |
| TPMS/공조 | `REST Baseline 유산` · ``TpmsPressure*`` | 개발 메모 톤 |
| 위치 empty | `취침(추론) · Location 수신` | 영문 필드명 |
| lifecycle 뱃지 | `Telemetry 대기` / `Telemetry 단절` | 제품명 노출(허용 가능하나 한글화 여지) |
| REST 사유 | `Wake 후 쿨다운 REST` | 내부 정책 용어 |
| 툴바 Provider | `Tesla Fleet API` / `Mock Provider` | 인프라·환경 라벨 — 법인 관리자에게 “이 차 상태”가 아님 |
| 툴바 갱신 | `갱신: 2026. 7. 13. 오전 10:41:04` | 절대시각만 있고 **무엇을** 갱신했는지 불명 · Summary의 상대시각과 중복 |
| 상단 식별·상태 | 툴바·헤더 카드·제원 접힘에 **번호·모델·트림·색·상태** 반복 | 스크롤 전 시야에 같은 정보가 3~4회 → 업무 질문(“지금 문제인가?”)보다 **무엇이 이 차인지**만 반복 |
| Summary SOC | 숫자만 `24.2%` | 목록의 `BatteryProgressBar`와 불일치 · 한눈에 잔량 비교 어려움 |
| Summary 충전 | 항상 `미연결` 등 뱃지 | 평시 노이즈 — “연결됨/충전 중”일 때만 의미 있음 |
| Summary 잠금 | 항상 `잠김`/`잠금 해제` | 정상(잠김)은 상단 고정 가치가 낮음 · 해제는 이슈/보안 구역으로 충분 |
| Summary 신선도 | `데이터 3시간 전 · 실시간` + `마지막 신호: 3시간 전 (절전…)` | **같은 시각을 두 줄**로 반복 |

DB/동작은 유지. **바꾸는 것은 표현 계층**이다.

## 4. To-Be — 표현 사전 (1차 화면)

### 4.1 운행 상태 (ASLEEP / 추론 포함)

| 조건 | 1차 표시 (확정 톤) | 보조 한 줄 (선택) | 운영·자세히 (기술) |
|------|-------------------|-------------------|-------------------|
| `status=ASLEEP` (+ 추론 여부 무관하게 1차는 동일) | **주차 (절전)** | **마지막 신호: N분/시간 전** (절전 모드) | `isAsleepInferred`이면 「신호 공백으로 절전으로 표시」 등 |
| `ONLINE` | **운행 가능** 또는 **온라인** (기존 톤 정리) | 마지막 신호 시각 | 소스 TELEMETRY/REST |
| `OFFLINE` | **연결 끊김** | 마지막 신호 | |
| `WARNING` / `ALERT` | **주의** / **경고** | 이슈 요약으로 유도 | |

초안 수정안 반영:

- ~~취침 (추론)~~ → **주차 (절전)**
- ~~Telemetry 공백으로 취침을 추론했습니다 · 4시간 전~~ → **마지막 신호: 4시간 전 (절전 모드)**

목록·지도 마커 라벨도 가능하면 동일 사전을 쓴다 (`취침 중` → `주차(절전)` 검토).

### 4.2 연동·configSynced 경고

| 조건 | 1차 표시 | 톤 |
|------|----------|-----|
| 구독 활성 + **최근 Telemetry 수신 있음** + `configSynced===false` | **실시간 수신 중** (경고 카드 **숨김** 또는 정보: 「차량 설정 반영 확인 중 — 수신은 정상」) | info / 없음 |
| 구독 활성 + **수신 오래됨/없음** + `configSynced===false` | **실시간 데이터 지연 가능** — 「다시 연결」 유도 | warning |
| 구독 비활성 / 단절 | **실시간 연동 꺼짐** | warning |
| 타임라인 config 단계 | 라벨 **실시간 설정** · 상세: `반영됨` / `반영 대기` (영문·필드명 금지) | `done`은 수신 성공 또는 synced true면 완료 처리 권장 |

필드명 `configSynced false` 뱃지 **1차 화면에서 제거**.

#### 4.6.1 「Tesla Fleet API」/「Mock Provider」뱃지

| 항목 | 내용 |
|------|------|
| **의미** | 앱 환경 변수 `VEHICLE_DATA_PROVIDER`(`tesla` \| `mock`)를 사람이 읽기 쉬운 라벨로 바꾼 것. **차량 상태·연동 건강도와 무관**하다. |
| **관리자 관점** | “이 차가 Tesla API로 연결됐다/안 됐다”처럼 읽히기 쉬움. 실제로는 배포 설정 표시에 가깝고, 연동 문제는 lifecycle·운영 카드가 담당한다. |
| **Mock 노출** | 운영/스테이징에서 `Mock Provider`가 보이면 데모·오해 소지가 큼. 상세 1차 화면에 둘 이유가 약하다. |

**권고 (채택 제안): 차량 상세 툴바에서는 Provider 뱃지를 제거한다.**

| 선택지 | 평가 |
|--------|------|
| **A. 상세에서 제거** (권고) | 1차 원칙(업무 결과만)에 맞음. Provider는 **설정·연동 화면** 또는 「기술 상세」에만 두면 충분. |
| B. 유지 | 개발·QA가 모드를 한눈에 볼 때 유용하나, 법인 관리자 상세에는 노이즈. |
| C. 상세만 숨기고 목록/대시보드 툴바는 유지 | 상세 집중도↑. 화면 간 불일치는 감수. |

목록·대시보드·지도 툴바의 동일 뱃지는 **본 요구의 필수 범위 밖**(상세만 우선). 일관성을 원하면 같은 Phase에서 일괄 제거 검토.

#### 4.6.2 「갱신: YYYY. M. D. 오전/오후 h:mm:ss」

| 항목 | 내용 |
|------|------|
| **데이터 기준** | 최신 `VehicleSnapshot.lastUpdatedAt`. 텔레메트리·REST 등으로 **스냅샷 행이 새로 쓰인 시각**. |
| **아닌 것** | (1) 「새로고침」버튼을 누른 시각 — 아님. (2) 화면을 연 시각 — 아님. (3) Tesla 서버 시각 전용 필드 — 아님. (4) Summary의 `데이터 N분 전`과 **동일 계열이지만**, Summary는 `lastTelemetryAt ?? lastUpdatedAt` 우선·상대시각. |
| **관리자 관점** | 라벨이 「갱신」만이라 *무엇을* 갱신했는지 모호. 절대시각(초 단위)은 절전·공백 상황과 함께 두면 Summary 상대시각과 **중복·경쟁**. |

**권고 (채택 제안): 상세 툴바에서는 「갱신: 절대시각」을 기본 미표시한다.**

| 선택지 | 평가 |
|--------|------|
| **A. 툴바에서 제거** (권고) | Summary의 `데이터 N분 전` / 절전 시 `마지막 신호: …`가 신선도 역할을 이미 수행. 툴바는 번호·모델·새로고침만. |
| B. 유지하되 라벨 변경 | 예: `데이터 기준: …` — 의미는 나아지나 1차 화면 밀도는 여전. |
| C. 상대시각으로 교체 | Summary와 문구가 겹침 → 둘 중 하나만. |
| D. 기술 상세·운영 카드로 이동 | 절대시각이 진단에 필요할 때. 이미 운영 카드에 실시간/상세 조회 시각이 있음. |

「새로고침」버튼은 유지(목록 registry 동기화 + DB 재조회). 버튼과 「갱신」시각을 동일시하지 않도록, 시각을 남길 경우에도 **버튼 클릭 시각으로 오해되지 않게** 라벨을 명확히 할 것.

**§4.7 권고 반영 완료 (UX2-11, 2026-07-13)** — 헤더 신원 카드 제거 · 제원 접힘 `VIN ···`만.

#### 4.7.1 비개발자 차량 관리자 관점

관리자가 상세에 들어오는 순간 필요한 답:

1. **이 차가 맞는가?** → 번호(표시명) 한 번이면 충분. 모델·연식은 보조 한 줄.
2. **지금 조치가 필요한가?** → 운행 상태 · SOC · 충전 · 잠금 · 이슈 수. (Summary가 담당)
3. **어디에 있는가 / 배터리·보안 상세는?** → 아래 지도·배터리·잠금 카드로 스크롤.
4. **연동이 끊겼는가?** → 문제일 때만 경고·운영 카드. 평시 헤더에 연동·트림·색 뱃지 나열은 산만.
5. **VIN·색·SW가 궁금한가?** → 빈도 낮음 → 제원 접기 안에만.

현재는 1번을 툴바·헤더·제원에서 반복하고, 2번(상태)도 Summary와 헤더에서 반복한다.  
**효율:** 동일 스캔 비용으로 새 정보가 거의 없음.  
**직관:** “어디를 보면 되는가?”가 흐려짐 — sticky Summary가 있어도 그 위·아래에 같은 번호·모델이 있으면 시선이 분산된다.

#### 4.7.2 배치 원칙 (채택 제안)

| 원칙 | 내용 |
|------|------|
| **한 사실 = 한 자리** | 번호·모델·VIN·트림·색·운행상태는 각각 **대표 구역 하나**. 다른 구역은 링크·접기로만 참조. |
| **위 = 상황, 아래 = 신원 상세** | 상단 1뷰포트: *상황(Summary)*. 제원·VIN은 접힌 「제원·식별」. |
| **헤더 카드는 역할이 없으면 제거** | 툴바 + Summary만으로 1·2번이 충족되면 **카드2(대형 신원 카드)는 폐지 또는 VIN·복사용 한 줄로 축소**. |
| **뱃지 남발 금지** | 헤더에 운행상태 + 연동 + 트림 + 색 + 충전을 한 줄에 두지 않음. 충전·연동은 Summary 또는 운영에만. |
| **제원 접힘 미리보기** | 접힌 상태에서 모델·트림·색을 또 쓰지 않음. 예: `VIN ···4742` 또는 `제원 보기`만. 펼치면 모델·트림·색·VIN·SW 전부. |

#### 4.7.4 선택지 비교

| 안 | 내용 | 평가 |
|----|------|------|
| **A. 헤더 카드 제거 + 제원 접힘 축소** (권고) | 툴바=식별, Summary=상황, 제원=정적 상세 | 중복 최소·VD IA·업무 효율에 가장 부합 |
| B. 툴바에서 모델 줄 제거, 헤더만 식별 | 제목이 번호만 → 헤더 유지 이유 생김 | 헤더가 커져 상황(Summary)보다 신원이 위에 남기 쉬움 |
| C. Summary에 번호까지 넣고 툴바 단순화 | sticky에 식별+상황 | 가능하나 툴바와 sticky 이중 제목 위험 |
| D. 현상 유지 + 문구만 정리 | 카피 UX2만 | **배치 중복은 해소되지 않음** |

평시에 헤더 뱃지로 두면 상태칩과 경쟁한다.

- **문제일 때:** Summary 아래 경고 카드 또는 운영 카드만 (기존 guidance).
- **평시 READY:** 헤더에 lifecycle 뱃지 비표시 (이미 `shouldShowLifecycleBadge` 취지와 동일).

#### 4.8.3 잠금 — 상단 고정이 필요한가?

| 질문 | 의견 |
|------|------|
| 왜 상단에 있었나? | VD-1 Summary가 “이상 여부”에 잠금을 넣어, sticky에서 보안을 한눈에 보려 한 것. |
| `잠김`을 sticky에 둘 가치? | **낮음.** 법인 플릿에서 주차 중 잠김은 **정상 기본값**. 매번 보이면 신호 대비 노이즈. |
| `잠금 해제`는? | **조치 필요** → Summary 상시 문구보다 **이슈 태그·잠금·개폐 카드**가 맞음. 이미 `collectIssueTags`에 「잠금 해제」가 있음. |

**권고: Summary에서 잠금 문구 전부 미노출** (요청과 동일).  
- 정상 잠김 → 표시 안 함.  
- 해제 → `이슈 N` / 보안 섹션으로만.  
상단에 “잠김”을 남겨 두면 절전·SOC·이슈와 **시선 경쟁**만 한다.

| 권고 | **채택.** UX2-14와 충돌 없음(문구 칩 ≠ 의도적 퀵타일) |
| 트렁크 | 「트렁크」→ **리어 우선**, 프렁크는 이슈 태그로 보완 |

| 항목 | 의견 |
|------|------|
| 권고 | **채택.** 타이어를 상황 블록(보안·건강)에 묶음. 다이어그램 소형·출처 한 줄 |

#### 4.9.7 선택지

| 안 | 내용 | 평가 |
|----|------|------|
| **A. 요청안 + 하단 공조 제거·보안 축소** (권고) | 중복 최소 | 업무 효율↑ |
| B. 요청안만 추가, 하단 유지 | 구현 빠름 | 동일 정보 반복 |
| C. 맵만 2열, 퀵타일·TPMS는 하단 | 상단 가볍게 | 요청 미충족 |

### 4.10 「현재 위치」카드 — 지도 하단 캡션·인근 충전소 (후속 의견)

As-Is (상세 `/vehicles/[id]` 우측 카드):

- 지도 아래(또는 선택 패널): `TESLA-214742` · `Model Y · RWD` · `주차 (절전)` · **상세 보기**  
  → `VehicleMap` 선택 차량 카드(`SelectedVehicle` 계열). 목록/전체 지도용 UI가 상세에 그대로 노출됨.
- 페이지 하단 별도 카드: **인근 충전소** 목록·empty.

#### 4.10.2 인근 충전소를 지도 아래로

| 항목 | 의견 |
|------|------|

| 밀도 | 우측 카드가 길어질 수 있음 → 목록 **상위 3~5곳**만, 더보기는 불필요 시 생략 |
