# 차량 관제 시스템(FMS) 요구사항 문서

## 1. 프로젝트 개요
- 차량 관제 시스템(FMS, Fleet Management System) 개발을 시작한다.
- 개발은 `Cursor AI`를 활용한 1인 개발 방식으로 진행한다.
- 초기 단계에서는 필요한 기능 위주로 MVP를 우선 개발한다.
- 1차 목표는 스타트업 데모데이에서 공개 및 시연 가능한 수준의 완성도를 확보하는 것이다.
- 이후 투자 유치가 이루어지면 기능 확장과 사업 확대를 추진한다.

## 2. 개발 원칙
- 1인 개발 환경에 맞게 현실적으로 구현 가능한 범위를 우선한다.
- 복잡한 기능보다 시연에 필요한 핵심 기능을 먼저 완성한다.
- 초기에는 운영과 유지보수가 쉬운 구조를 우선 채택한다.
- 빠르게 만들고 검증한 뒤, 필요한 기능을 단계적으로 확장한다.

## 3. MVP 목표
초기 MVP의 목표는 PC 웹 대시보드에서 전체 차량 현황을 확인하고 관리할 수 있는 최소 기능을 제공하는 것이다.

핵심 목표는 다음과 같다.
- 차량 목록을 조회할 수 있어야 한다.
- 각 차량의 현재 위치를 확인할 수 있어야 한다.
- 차량의 기본 상태 정보를 확인할 수 있어야 한다.
- 관리자 관점에서 전체 차량을 한 화면 또는 몇 단계 이내에서 관리할 수 있어야 한다.

## 4. 대상 사용자
초기 주요 사용자는 차량을 운영하거나 관리하는 관리자이다.

예상 사용자 예시는 다음과 같다.
- 법인 차량 운영 담당자
- 관제 담당자
- 차량 상태를 모니터링해야 하는 운영 관리자

## 5. 시스템 범위
시스템의 큰 구조는 PC 웹 대시보드 중심으로 설계한다.

초기 범위는 다음과 같다.
- 전체 차량 목록 조회
- 차량별 상세 정보 조회
- 차량 위치 정보 확인
- 차량 상태 정보 확인
- 관리자용 기본 관리 화면 제공

## 6. 차량 데이터 수집 방향
차량의 데이터(위치, 차량 상태 등)는 별도의 전용 단말기 없이 수집하는 방식을 우선 검토한다.

수집 방식은 데이터 종류에 따라 현실적으로 구분한다.
- **텔레매틱스 데이터(배터리, 충전, 주행가능거리, 경고등, 시동 등)**: 제조사(OEM) 클라우드 API를 통해 차량 소유자 계정 인증(OAuth) 기반으로 수집한다. 이 데이터는 스마트폰 앱만으로는 확보할 수 없다.
- **위치(GPS) 데이터**: OEM API로 함께 제공받거나, 필요 시 스마트폰 앱을 통한 GPS 전송으로 보완한다.

초기에는 하드웨어(OBD/IoT 단말기) 설치가 필요한 방식보다 OEM API 기반 소프트웨어 수집 방식을 우선한다.

### 6.1 데이터 접근 리스크와 대응
- OEM별로 공식 API의 존재 여부와 개방 수준이 크게 다르므로, 착수 전 데이터 접근 가능성을 반드시 검증한다.
- 실제 API 연동이 지연되거나 승인이 늦어질 경우를 대비해, **데모용 Mock(시뮬레이션) 데이터**를 항상 폴백으로 준비한다.
- 데모데이 시연은 실제 API 또는 Mock 중 안정적인 쪽으로 유연하게 전환할 수 있어야 한다.

## 7. 초기 타깃 차종 전략
초기 개발 대상은 전기차 중심으로 하며, MVP 단계에서는 **테슬라(Tesla) 1종으로 확정**한다.

검토했던 차종/브랜드와 데이터 접근성 판단은 다음과 같다.

| 브랜드 | 공식 데이터 접근 | 판단 |
|--------|------------------|------|
| **테슬라** | 공식 Fleet API + 소유자 OAuth 동의로 개인 차량 접근 가능 | **채택** (1인 개발자가 현실적으로 연동 가능) |
| 현대차 | 공식 공개 개발자 API 사실상 부재, 앱 전용 | 제외 (비공식 접근은 약관 위반 소지) |
| 기아차 | 공식 공개 개발자 API 사실상 부재, 앱 전용 | 제외 (동일) |
| BYD | 국내 데이터 접근 경로 매우 제한적 | 제외 |

테슬라 선정 사유는 다음과 같다.
- 공식 Fleet API가 존재하여 데이터 접근이 합법적·안정적이다.
- 소유자 OAuth 동의만으로 개인 차량 데이터를 받을 수 있어 1인 개발에 적합하다.
- 위치·배터리·충전·주행상태 등 관제에 필요한 핵심 데이터를 제공한다.

유의 사항:
- 테슬라 Fleet API는 등록 절차와 사용 과금이 있을 수 있으므로 착수 전 조건을 확인한다.
- 실제 연동이 어려운 구간은 Mock 데이터로 대체하여 데모 일정을 보호한다.
- 향후 다차종 확장을 고려해 데이터 수집부를 차종 독립적인 구조로 설계한다.

## 8. 핵심 기능 요구사항
초기 MVP에 필요한 핵심 기능은 다음과 같다.

- 차량 등록 및 목록 관리
- 차량별 현재 위치 조회
- 차량별 상태 정보 조회
- 전체 차량 현황 대시보드
- 차량 상세 화면
- 최근 데이터 수신 시간 표시
- 비정상 상태 차량 식별

차량 상태 정보 예시는 다음과 같다.
- 온라인/오프라인 상태
- 배터리 상태 또는 주행 가능 관련 정보
- 최근 위치 업데이트 시간
- 기타 연동 가능한 기본 차량 상태 값

## 9. 비기능 요구사항
- 웹에서 빠르게 확인 가능한 직관적인 UI를 제공해야 한다.
- 데모 시연 중 문제 없이 동작할 정도의 안정성을 확보해야 한다.
- 운영 비용은 가능한 낮게 유지해야 한다.
- 빠른 수정과 확장이 가능한 단순한 구조를 지향한다.

### 9.1 법적·개인정보 요건
차량 위치·운행 데이터는 개인정보 및 위치정보에 해당할 수 있으므로 아래를 준수한다.

- 차량 위치·운행 데이터는 「위치정보의 보호 및 이용 등에 관한 법률(위치정보법)」과 「개인정보보호법」의 적용 대상이 될 수 있다.
- 차량 소유자·운전자의 위치정보 및 개인정보 수집·이용에 대한 사전 동의를 확보해야 한다.
- OEM API 연동 시 소유자 계정 인증(OAuth) 동의 절차를 준수한다.
- **데이터 모델**: FMS 고객(User) · 테슬라 계정(TeslaAccount) · 차량(Vehicle) 계층 구조 및 연동 해제 정책은 [requirements-user-db.md](./requirements-user-db.md)를 따른다.
- **감사 로그**: Tesla/FMS 민감 API의 기록·마스킹·추적 정책은 [requirements-log-db.md](./requirements-log-db.md)를 따른다.
- **데모용 가상 차량**: 실차 `sleep` 상태 대응을 위한 `차량 추가(가상)` 시드 정책은 [requirements-virtual-vehicle-seeding.md](./requirements-virtual-vehicle-seeding.md)를 따른다.
- 서비스 상용화 단계에서는 위치정보사업/위치기반서비스사업 신고 필요 여부를 검토한다.
- 개인정보 처리방침을 마련하고, 데이터 접근 권한과 보관 기간을 정의한다.
- **데모 단계 완화책**: 자가 소유 차량 또는 Mock 데이터로 시연하여 법적 리스크를 최소화한다.

## 10. MVP 제외 범위
초기 단계에서는 아래 기능을 제외하거나 후순위로 둔다.

- 다수 차종 동시 지원
- 전용 OBD 또는 IoT 단말기 개발
- 정교한 배차 기능
- 정산/과금 기능
- 복잡한 권한 관리
- 고급 통계 및 리포트
- 모바일 앱 정식 출시
- 대규모 실시간 관제 아키텍처

## 11. 성공 기준
1차 목표 달성 기준은 다음과 같다.

### 11.1 정성적 기준
- 타깃 차종(테슬라)에 대해 차량 데이터를 정상 수집할 수 있다. (실 연동 또는 Mock)
- PC 웹 대시보드에서 전체 차량 현황을 확인할 수 있다.
- 차량 위치 및 기본 상태를 시연할 수 있다.
- 데모데이에서 전체 서비스 흐름을 끊김 없이 설명할 수 있다.

### 11.2 정량적 지표
측정 가능한 목표치는 다음과 같다. (수치는 데모 환경 기준이며 착수 후 조정 가능)

| 지표 | 목표치 |
|------|--------|
| 대시보드에 동시 표시 가능한 차량 수 | 최소 10대 이상 |
| 차량 위치·상태 갱신 주기 | 5분 이내 (폴링 기준) |
| 대시보드 첫 화면 로딩 시간 | 3초 이내 |
| 이상/미운행 차량 식별 정확도 | 정의된 규칙에 따라 100% 반영 |
| 데모 시연 시나리오 | 3종 무중단 수행 |
| 데이터 소스 전환(실연동↔Mock) | 30초 이내 전환 가능 |

## 12. 일정 및 마일스톤
데모데이를 기준일(D-day)로 두고 역산하여 단계별 목표를 관리한다. (구체적 날짜는 데모데이 확정 시 기입)

| 단계 | 기간(예시) | 목표 |
|------|-----------|------|
| M1. 기반 구축 | D-day 4~5주 전 | 프로젝트 스캐폴딩, DB 스키마, Mock 데이터 |
| M2. 핵심 화면 | D-day 3주 전 | 대시보드·차량 목록·상세·지도 UI |
| M3. 데이터 연동 | D-day 2주 전 | 테슬라 Fleet API 연동, 폴링 동기화 |
| M4. 안정화 | D-day 1주 전 | 이상/미운행 식별, 데모 시나리오 리허설 |
| M5. 데모데이 | D-day | 시연 및 피드백 수집 |

> 일정 리스크 발생 시, 실 API 연동(M3)보다 Mock 기반 시연 완성도를 우선한다.

## 13. 가정 및 제약
- **인력**: 1인 개발자 + Cursor AI, 별도 팀 없음
- **기간**: 데모데이까지 약 4~5주 (파트타임 가능성 고려)
- **예산**: 무료/저가 티어 중심, 유료 과금 최소화
- **차량 규모**: 데모 기준 소수(10대 내외), 대규모 실시간 부하는 범위 외
- **데이터 소스**: 테슬라 Fleet API 또는 Mock, 그 외 차종 미지원
- **환경**: PC 웹(관리자) 우선, 모바일·다국어 미지원
- **네트워크**: 데모 현장 인터넷 불안정 가능성 → 로컬/Mock 폴백 대비

## 14. 리스크 관리

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 테슬라 Fleet API 승인·과금 지연 | 실 데이터 연동 불가 | Mock 데이터로 데모 완성, 환경변수로 전환 |
| 데모 현장 네트워크 불안정 | 시연 중단 | 로컬 실행/Mock 캐시 준비 |
| 1인 개발 일정 지연 | 기능 미완성 | P0 기능 우선, P1·P2 과감히 후순위화 |
| 개인정보·위치정보 규제 | 상용화 제약 | 데모는 자가 차량/Mock, 상용화 전 법률 검토 |
| 범위 확장(스코프 크립) | 완성도 저하 | MVP 제외 범위(§10) 엄격 준수 |

## 15. 용어 정의

| 용어 | 설명 |
|------|------|
| FMS | Fleet Management System, 차량 관제/관리 시스템 |
| MVP | Minimum Viable Product, 핵심 기능만 갖춘 최소 제품 |
| Deviceless | 전용 단말기 없이 OEM API 등으로 데이터를 수집하는 방식 |
| OEM API | 차량 제조사가 제공하는 차량 데이터 연동 인터페이스 |
| OAuth | 소유자 계정 인증·동의 기반 데이터 접근 권한 위임 방식 |
| Mock 데이터 | 실제 연동 대신 사용하는 시뮬레이션 차량 데이터 |
| 스냅샷 | 특정 시점의 차량 최신 상태(위치·배터리 등) 데이터 |
| TeslaAccount | FMS에 연동된 테슬라 OAuth 계정 — 토큰·차량 목록의 단위 ([requirements-user-db.md](./requirements-user-db.md)) |
| 연동 해제(Unlink) | FMS 관제 대상에서 차량·계정을 제외하는 Soft Delete — Telemetry 구독 선해제 필요 |
| AuditLog | 사용자 행위 중심의 감사 로그 — 누가 어떤 민감 작업을 수행했는지 기록 |
| ApiCallLog | Tesla/FMS HTTP 요청·응답 중심의 통신 로그 — request/response·status·duration 기록 |
| Geofencing | 지도상 관심 지역 설정 및 진출입 감지 기능 |
| 폴링 | 일정 주기로 데이터를 반복 조회해 갱신하는 방식 |
