## 1. 배경

- CI-A~B(초안): 실내·실외·변속을 다이어그램에 merge · 퀵타일 4칸.  
- CI-C: sample HTML 차체·타이어 비주얼.  
- **CI-D (확정):** 실내·실외·변속을 **다이어그램에서 원복** → **3×2 서브카드**. 다이어그램은 TPMS 전용.

## 2. To-Be 레이아웃

```
상태 · SOC · (충전 카드)
┌ 잠금 │ 변속 │ 감시모드 ┐   ← 3×2 퀵타일
│ 공조 │ 실내 │ 실외     │
─── 타이어 · 차체 ───
  PSI|■  ┌글래스루프차체┐  ■|PSI
  PSI|■  └─────────────┘  ■|PSI
```

| 정보 | 위치 |
|------|------|
| 잠금 · 변속 · 감시모드 · 공조 · 실내 · 실외 | **3×2 퀵타일** |
| TPMS | 「타이어 · 차체」다이어그램 |

## 3. 3×2 퀵타일 (CI-D)

| # | 라벨 | 데이터 | 표시 예 |
|---|------|--------|---------|
| 1 | 잠금 | `locked` | 잠김 / 해제 |
| 2 | 변속 | `shiftState` | P / R / N / D (툴팁: 주차 등) |
| 3 | 감시모드 | `sentryMode` | 활성 / 비활성 |
| 4 | 공조 | `climateOn` | ON / OFF |

## 10. 「실시간 차량 정보」카드 평가 (2026-07-14)

FMS 차량 상세 컨셉 기준:  
[requirements-vehicle-detail-ui.md](./requirements-vehicle-detail-ui.md) IA(*지금 문제인가? → 어디? → 충전·보안 → 연동 → 제원*) ·  
[requirements-vehicle-detail-ui2.md](./requirements-vehicle-detail-ui2.md) (*결과 먼저 · 한 사실=한 자리 · 관리자 언어*).

평가 대상은 좌측 카드 전체 스택이다, TPMS만이 아니다.

```
① 제목 「실시간 차량 정보」
② 상태 · SOC바 · 주행가능 km · 이슈 · 신선도 / 절전 시 마지막 신호
③ 충전 서브카드 (조건부, CC)
④ 3×2 퀵타일 (CI-D)
⑤ 「타이어 · 차체」TPMS (CI-C · CI-C2 · **CI-C3**)
```

우측 「현재 위치」는 1뷰포트 페어로 평가하되, 본 절의 개선 ID는 좌측 중심.

### 10.1 종합 판정

| 축 | 점수(5) | 한 줄 |
|----|:-------:|------|
| **컨셉 정합** | 4 | 상단이 “상황 SoT”로 자리 잡음. 관리자 질문 1~3(문제·위치·충전/보안)에 맞는 밀도 |
| **데이터 선별** | 3.5 | 관제 핵심(SOC·잠금·감시·공조·TPMS)은 맞음. 개폐(문·트렁크)가 상단에서 빠짐 |
| **배치·정보구조** | 4 | 세로 계층(상태→이벤트→타일→타이어) 명확. 하위 배터리 카드와 SOC 중복은 남음 |
| **시각화** | 4 | SOC바·충전카드·글래스루프 TPMS는 제품감↑. 퀵타일은 텍스트 위주라 이상 시인성은 보통 |
| **신뢰·신선도** | 3.5 | 절전·출처 힌트 좋음. TPMS 「마지막 상세 조회」와 퀵타일(텔레메트리 가능값)의 **시간축이 혼재** |

**한 줄 요약:** FMS 상세의 “스크롤 없이 차량 상황 읽기” 목적은 **대체로 충족**.  
남는 과제는 **개폐 시인성**, **타일 vs 다이어그램의 신선도 통일**, **이상 상태 시각 강조**, **하단과의 중복 정리**.

### 10.2 잘 맞는 점 (유지 권고)

| 영역 | 평가 |
|------|------|
| **IA 상단 2열** | 상황 \| 위치 — VD *어디?*를 스크롤 없이 제공. UX2-16과 정합 |
| **상태 스트립** | 운행상태 + SOC바 + 주행km + 이슈 — *지금 문제인가?*의 1차 답 |
| **충전 카드 (CC)** | 이벤트성만 노출 — UX2 “평시 노이즈 줄이기”와 맞음 |
| **퀵타일 CI-D 구성** | 잠금·변속·감시·공조·온도 — 관제·원격에서 자주 묻는 축. 필드명 미노출 |
| **타일 순서** | 잠금→변속(운행 맥락)→감시→공조→온도 — 보안→동력→편의 흐름이 납득됨 |
| **TPMS 비주얼 (CI-C)** | 공간적 TPMS는 목록/텍스트보다 운영 직관에 유리. 정상/주의 이중 단서(수치+뱃지+펄스) |
| **한 자리 (부분)** | 감시·변속을 퀵타일 SoT로 두고 하단 중복 제거한 방향은 UX2 원칙과 일치 |

### 10.3 아쉬운 점 · 갭

#### A. 데이터 · 관제 우선순위

| 갭 | 설명 | 심각도 |
|----|------|:------:|
| **문·트렁크 상단 부재** | UX2-F 당시 퀵타일에 문(종합)·트렁크가 있었음. CI-D에서 감시·변속으로 교체 → **개폐 이상은 이슈 링크·하단 상세까지 스크롤**해야 함. 법인 관리자가 “문 열림?”을 1초안에 보려면 약함 | 중 |
| **변속 vs 개폐 트레이드오프** | `P/D`는 Telemetry에서 자주 오지만, FMS 페르소나에선 **개폐·잠금**이 변속보다 조치 긴급도가 높은 경우가 많음. 변속이 2번 자리는 UX적 무게가 큼 | 중(논쟁) |
| **이슈 태그와 퀵타일 연결 약함** | 「이슈 N」은 `#vehicle-security`로 점프하나, **어떤 타일이 문제인지**(잠금 해제·감시 off 등) 타일 자체에 연결되지 않음 | 중 |
| **실내·실외 정밀도** | `N.N°C`는 개발자 느낌. 관제는 대개 정수 °C면 충분 (표시만 반올림) | 낮 |

#### B. 배치 · 밀도

| 갭 | 설명 | 심각도 |
|----|------|:------:|
| **좌측 카드 세로 길이** | 상태+충전+6타일+대형 TPMS → xl에서도 좌·우 높이 불균형·스크롤 유도. “1뷰포트 상황” 목표와 긴장 | 중 |
| **「타이어 · 차체」제목 vs 실내용** | 차체 일러스트는 있지만 표시 정보는 **타이어뿐**. 제목이 기대(차체 상태)를 과장할 수 있음 → 「타이어」또는 「타이어 공기압」검토 | 낮 |
| **하단 배터리·SOC 중복** | 상단 프로그레스와 하단 `BatteryHealthGauge`가 SOC/거리를 다시 말함. VD에서도 “중복 최소화” 지적 | 중(페이지 전체) |
| **좁은 폭 3열** | 라벨 「감시모드」+값 2줄은 가능하나, 긴 번역·큰 폰트에서 넘침 위험 | 낮 |

#### C. 시각화 · 시인성

| 갭 | 설명 | 심각도 |
|----|------|:------:|
| **퀵타일 = 라벨+텍스트** | 정상/이상이 **톤(보더색)**에만 의존. 잠금「해제」·감시「비활성」이 더 강하게 튀지 않음 (아이콘·강조 값 색 부족) | 중 |
| **변속 토큰만 노출** | `P`/`D`는 운전자 UX. 툴팁에만 한글 — 모바일 툴팁 약함. 기본 표시를 `P · 주차` 또는 `주차`로 할지 선택지 | 낮~중 |
| **TPMS 출처 vs 신선도** | 다이어그램은 REST 유산 출처가 흔함. 퀵타일 온도·변속은 Telemetry일 수 있음 → **카드 안 시간축이 둘** | 중 |
| **TPMS 없을 때 empty** | PSI 전부 null이면 구역 숨김 → 절전·미구독 시 “타이어를 아직 못 봤는지” 안내가 없음 | 낮 |
| **샘플 대비** | sample은 주의 앰버 중심. 임계 `isLowTpms`(기본 35 PSI)과 실차 atm→PSI 환산 검수 의존 | 검수 |

#### D. 컨셉 리스크 (표현)

| 갭 | 설명 |
|----|------|
| 제목 「실시간」 | REST 유산·절전 공백도 이 카드에 옴. UX2에서 이미 “부제(신선도)로 보완 OK” — **현 유지 가능**. 다만 TPMS stale과 조합되면 “실시간처럼 보이는데 타이어는 오래됨” 인지 부조화 |
| 관제 vs 편의 | 실내·실외·공조는 운영(과열·프리컨디션)에도 쓰이지만, **개폐보다 우선 배치**된 인상은 법인 FMS보다 개인 앱에 가깝다는 비판 가능 |

### 10.4 개선안 (우선순위 · CI-E 후보)

코드 미착수. 채택 시 ID를 CI-11+로 승격.

| ID | 개선 | 권고 | 우선 | 기대 효과 |
|----|------|:----:|:----:|-----------|
| **CI-11** | 퀵타일 **개폐 시인성**: (안A) `문(종합)` 또는 `트렁크`를 1칸 복귀 — 후보 교체: 변속을 배터리 구역으로 되돌리거나, 감시·변속 중 하나와 스왑. (안B) 상단 상태 스트립에 「문 개방」칩만 이슈와 동의어로 유지 | **안A 검토** | P1 | 보안 IA 복구 |
| **CI-12** | 이상 타일 강조: `해제`·문개방·저온/고온 임계 시 **값 색 + 아이콘**(잠금/열림) — 색만 금지(접근성) | 채택 | P1 | 스캔 속도 |
| **CI-13** | 신선도 통일: 카드 상단 한 줄 + TPMS는 「타이어는 N전 상세조회」만 **값이 stale일 때** 보조. 중복 출처 문구 축소 | 채택 | P1 | 신뢰 |
| **CI-14** | 변속 표시: 기본 `주차`/`주행`… 또는 `P 주차`. 영문 단독 금지 강화 | 채택 | P2 | UX2 |
| **CI-15** | 온도 표시 정수 °C (내부 소수 유지) | 채택 | P2 | 밀도 |
| **CI-16** | 구역 제목 「타이어 · 차체」→ 「타이어」또는 「타이어 공기압」 | 선택 | P2 | 기대치 |
| **CI-17** | TPMS empty: PSI 없을 때 접힌 안내 1줄(「타이어는 상세 조회 후 표시」) | 채택 | P2 | 공백 이해 |
| **CI-18** | 좌측 밀도: TPMS `max-height`/축소 또는 타일↔TPMS 접기(기본 펼침) — 맵 높이와 맞춤 | 선택 | P2 | 1뷰포트 |
| **CI-19** | 이슈→타일: 관련 퀵타일에 `ring`/뱃지(예: 잠금 해제 시 잠금 타일) | 채택 | P2 | 조치 연결 |
| **CI-20** | 하단 배터리 게이지와 상단 SOC **역할 분리** 문서화·UI(예: 하단은 건강·충전한도·odometer만) — 페이지 단위, CC/VD와 공조 | 검토 | P2 | 한 자리 |

#### 퀵타일 교체안 (CI-11 상세)

| 안 | 6칸 구성 | 장점 | 단점 |
|----|----------|------|------|
| **현행 CI-D** | 잠금·변속·감시·공조·실내·실외 | 온도·공조·변속 한눈에 | 개폐 상단 약함 |
| **E1** | 잠금·**문**·감시·공조·실내·실외 | 개폐 복귀 | 변속·트렁크 약함 |
| **E2** | 잠금·**문**·**트렁크**·감시·공조·실내 | UX2-F에 근접 | 온도·변속 하단화 |
| **E3** | 잠금·변속·감시·**문**·실내·실외 | 절충 | 공조 off |

권고 절차: 실차 검수에서 “문 열림을 먼저 못 찾았다” 피드백이 나오면 **E1**을 CI-E로 착수. 피드백 없으면 현행 유지 + **CI-12·13**만 진행.
